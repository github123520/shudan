# 架构草案

## 目标

围绕一本书建立完整的“书单情报页”：

- 全量评语，不漏页
- AI 总结
- 高信噪比筛选
- 多本书共现找书单
- 基于书单网络推荐新书

## 推荐架构

```text
用户请求
  -> API 服务
  -> 查 PostgreSQL / 向量索引
  -> 返回结构化结果

后台抓取
  -> 任务队列
  -> 起点图抓取器
  -> 原始 HTML / 结构化结果
  -> PostgreSQL

异步分析
  -> AI 摘要
  -> 关键词提取
  -> 去模板/去简介搬运
  -> 书单画像
  -> 共现图更新
```

## 关键设计

### 1. 用户查询不直接触发现场抓取

前台只读数据库。抓取在后台排队执行。

原因：

- 页面抓取有延时约束
- 现场抓取会让移动端体验很差
- 同一本书和同一个书单会被反复复用

### 2. 书和书单都要独立建模

不是只存“某本书的评语列表”，而是至少有这三层关系：

- `books`
- `booklists`
- `booklist_books`

这样第 2 个核心能力“2~n 本书共现找书单”才做得起来。

### 3. 先保存原始结果，再做 AI 层

AI 总结必须是派生数据，不要把 AI 输出当事实源。

建议分层：

- 原始事实层：抓到的评语、书单、时间、关注数
- 清洗层：去重、切句、关键词、情绪、模板识别
- 智能层：摘要、画像、推荐分

## 任务拆分

### 阶段 1：单书抓全

- 输入 `bookId`
- 解析总书单数
- 计算总页数
- 分页抓取
- 输出 JSON

### 阶段 2：书单详情复用

- 从单书页里抽出全部 `booklistId`
- 数据库没有的书单再抓详情
- 已存在的书单低频刷新

### 阶段 3：共现索引

- 根据 `booklist_books` 建倒排
- 支持 `A AND B AND C` 查询
- 计算书单价值分和口味相似分

### 阶段 4：AI 能力

- 单书总结
- 争议点聚类
- 书单主口味画像
- 个性化推荐

## 为什么先做网页/PWA

- 后端和抓取是主体
- 移动端主要承担浏览、搜索、收藏
- 网页改动快，适合频繁调整排序和 AI 提示词
- 后续要 APK，直接套壳即可
